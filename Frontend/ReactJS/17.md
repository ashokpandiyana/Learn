# Chapter 17: Architecture & Folder Structure - In-Depth Explanation

## Why Architecture Matters

**Good architecture ensures:**
- ✅ Easy to find files
- ✅ Scalable as app grows
- ✅ Clear module boundaries
- ✅ Easy to onboard new developers
- ✅ Consistent patterns across codebase
- ✅ Easy to refactor and maintain

**Bad architecture results in:**
- ❌ "Where does this file go?"
- ❌ Circular dependencies
- ❌ Tight coupling
- ❌ Hard to test
- ❌ Slow to add features

---

## 17.1 Project Structure Patterns

### Pattern 1: Simple Structure (Small Apps)

**Best for:** Small projects (< 10 components)

```
src/
├── components/
│   ├── Header.jsx
│   ├── Footer.jsx
│   ├── Button.jsx
│   └── Card.jsx
├── pages/
│   ├── Home.jsx
│   ├── About.jsx
│   └── Contact.jsx
├── styles/
│   ├── globals.css
│   └── components.css
├── utils/
│   ├── helpers.js
│   └── constants.js
├── App.jsx
└── main.jsx
```

**Pros:**
- Simple and flat
- Easy to understand
- Quick to set up

**Cons:**
- Doesn't scale
- No clear organization
- Hard to maintain when it grows

### Pattern 3: Atomic Design

**Best for:** Design system-driven projects

```
src/
├── components/
│   ├── atoms/              # Smallest components (1 responsibility)
│   │   ├── Button/
│   │   │   ├── Button.jsx
│   │   │   ├── Button.test.js
│   │   │   ├── Button.stories.js
│   │   │   └── Button.module.css
│   │   ├── Input/
│   │   ├── Label/
│   │   ├── Icon/
│   │   └── Badge/
│   │
│   ├── molecules/          # Combinations of atoms
│   │   ├── FormField/      # Label + Input + Error
│   │   ├── SearchBar/      # Input + Button
│   │   ├── Card/
│   │   └── AlertMessage/
│   │
│   ├── organisms/          # Complex components
│   │   ├── Header/         # Logo + Navigation + SearchBar
│   │   ├── Footer/
│   │   ├── LoginForm/      # Multiple FormFields + Button
│   │   ├── ProductCard/
│   │   └── CommentSection/
│   │
│   ├── templates/          # Page layouts
│   │   ├── MainTemplate/
│   │   ├── DashboardTemplate/
│   │   └── AuthTemplate/
│   │
│   └── pages/              # Full pages
│       ├── HomePage/
│       ├── ProductPage/
│       └── CheckoutPage/
│
├── hooks/
├── services/
├── store/
└── utils/
```

**Atomic Design Hierarchy:**
```
Atoms (Button, Input)
  ↓ combine into
Molecules (FormField = Label + Input + Error)
  ↓ combine into
Organisms (LoginForm = FormFields + Button)
  ↓ combine into
Templates (AuthTemplate = Header + LoginForm + Footer)
  ↓ combine into
Pages (LoginPage = AuthTemplate + specific data)
```

## 17.3 Naming Conventions

### File Naming

**Components:**
```
✅ PascalCase (recommended)
Button.jsx
UserProfile.jsx
TodoList.jsx

✅ kebab-case (alternative)
button.jsx
user-profile.jsx
todo-list.jsx

❌ camelCase (avoid)
button.jsx
userProfile.jsx
```

**Non-Components:**
```
✅ camelCase for utilities, hooks, services
useAuth.js
userService.js
formatDate.js
validators.js

✅ UPPER_CASE for constants
API_URL.js
COLORS.js
```

**Test Files:**
```
Button.jsx
Button.test.js        # or Button.test.jsx

# Alternative
Button.jsx
__tests__/Button.test.js
```

### Component Naming

```javascript
// ✅ Good - Descriptive, clear purpose
function UserProfileCard() { }
function ProductListItem() { }
function ShoppingCartButton() { }
function AuthenticationModal() { }

// ❌ Bad - Too generic
function Card() { }
function Item() { }
function Button() { }
function Modal() { }

// Context: Good generic names are OK in shared components
// Button in shared/components is fine
// Card in shared/components is fine
```

### Variable Naming

```javascript
// ✅ Good - Clear and descriptive
const [isLoading, setIsLoading] = useState(false);
const [userList, setUserList] = useState([]);
const [selectedUserId, setSelectedUserId] = useState(null);

// ❌ Bad - Unclear
const [loading, setLoading] = useState(false);
const [data, setData] = useState([]);
const [id, setId] = useState(null);

// Event handlers
const handleSubmit = () => { };      // ✅ Good
const handleUserClick = () => { };   // ✅ Good
const onClick = () => { };           // ❌ Generic

// Boolean variables
const isActive = true;               // ✅ Good
const hasPermission = false;         // ✅ Good
const canEdit = true;                // ✅ Good
const active = true;                 // ❌ Not clearly boolean
```

## 17.4 Module Boundaries

### Import Rules

```javascript
// ✅ Good - Clear dependencies
// features/todos can import from:
import { Button } from '@/shared/components';
import { useDebounce } from '@/shared/hooks';
import { formatDate } from '@/shared/utils';

// ❌ Bad - Features shouldn't import from other features
import { UserAvatar } from '@/features/users';
// This creates tight coupling!

// ✅ Better - Move to shared if needed by multiple features
import { UserAvatar } from '@/shared/components';
```

**Dependency Direction:**
```
Pages
  ↓ can import
Features
  ↓ can import
Shared
  ↓ can import
Utils

❌ Never import upward!
Shared should NOT import from Features
Features should NOT import from Pages
```

### Absolute Imports

**Configure Path Aliases:**

```javascript
// jsconfig.json or tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@/components/*": ["src/components/*"],
      "@/features/*": ["src/features/*"],
      "@/shared/*": ["src/shared/*"],
      "@/utils/*": ["src/utils/*"],
      "@/hooks/*": ["src/hooks/*"],
      "@/types/*": ["src/types/*"]

  }
}

// Before

// After

```

### Service Layer Pattern

```javascript
// features/todos/services/todoService.js
import { apiClient } from '@/shared/api/client';

class TodoService {
  async getTodos(filters = {}) {
    const params = new URLSearchParams(filters);
    const response = await apiClient.get(`/todos?${params}`);
    return response.data;
  }
  
  async getTodoById(id) {
    const response = await apiClient.get(`/todos/${id}`);
    return response.data;
  }
  
  async createTodo(data) {
    const response = await apiClient.post('/todos', data);
    return response.data;
  }
  
  async updateTodo(id, data) {
    const response = await apiClient.patch(`/todos/${id}`, data);
    return response.data;
  }
  
  async deleteTodo(id) {
    await apiClient.delete(`/todos/${id}`);
  }
  
  async toggleTodo(id) {
    const response = await apiClient.patch(`/todos/${id}/toggle`);
    return response.data;
  }
}

export const todoService = new TodoService();
```

**Using Service in Hook:**
```javascript
// features/todos/hooks/useTodos.js
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { todoService } from '../services/todoService';

export function useTodos(filters) {
  const queryClient = useQueryClient();
  
  // Fetch todos
  const { data: todos, isLoading, error } = useQuery({
    queryKey: ['todos', filters],
    queryFn: () => todoService.getTodos(filters)
  });
  
  // Create todo
  const createTodo = useMutation({
    mutationFn: todoService.createTodo,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['todos'] });
    }
  });
  
  // Update todo
  const updateTodo = useMutation({
    mutationFn: ({ id, data }) => todoService.updateTodo(id, data),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['todos'] });
    }
  });
  
  // Delete todo
  const deleteTodo = useMutation({
    mutationFn: todoService.deleteTodo,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['todos'] });
    }
  });
  
  return {
    todos,
    isLoading,
    error,
    createTodo: createTodo.mutate,
    updateTodo: updateTodo.mutate,
    deleteTodo: deleteTodo.mutate
  };
}
```

**Using in Component:**
```javascript
// features/todos/components/TodoList/TodoList.jsx
import { useTodos } from '../../hooks/useTodos';
import { TodoItem } from '../TodoItem';

export function TodoList({ filters }) {
  const { 
    todos, 
    isLoading, 
    error, 
    deleteTodo, 
    updateTodo 
  } = useTodos(filters);
  
  if (isLoading) return <div>Loading...</div>;
  if (error) return <div>Error: {error.message}</div>;
  
  return (
    <ul>
      {todos.map(todo => (
        <TodoItem
          key={todo.id}
          todo={todo}
          onToggle={(id) => updateTodo({ id, data: { done: !todo.done } })}
          onDelete={deleteTodo}
        />
      ))}
    </ul>
  );
}
```

### API Client Setup

```javascript
// shared/api/client.js
import axios from 'axios';

const apiClient = axios.create({
  baseURL: process.env.REACT_APP_API_URL,
  timeout: 10000,
  headers: {
    'Content-Type': 'application/json'
  }
});

// Request interceptor
apiClient.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// Response interceptor
apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      // Redirect to login
      window.location.href = '/login';
    }
    return Promise.reject(error);
  }
);

export { apiClient };
```

## Summary of Chapter 17

**Key Architectural Patterns:**

1. **Feature-Based Structure** (Recommended)
   - Self-contained features
   - Clear boundaries
   - Easy to scale
   - Team-friendly

2. **Atomic Design**
   - Design system approach
   - Atoms → Molecules → Organisms → Templates → Pages
   - Great for component libraries

3. **Domain-Driven Design**
   - Business domain focus
   - Enterprise applications
   - Complex business logic

**Best Practices:**

```javascript
// ✅ Do's
✅ Group by feature, not by type
✅ Use absolute imports (@/features/*)
✅ Keep features independent
✅ Share via shared/ directory
✅ Use barrel exports (index.js)
✅ Consistent naming conventions
✅ Clear module boundaries

// ❌ Don'ts
❌ Mix feature code with shared code
❌ Create circular dependencies
❌ Import between features
❌ Deep relative imports (../../../../)
❌ Inconsistent file naming
❌ Mixing concerns in one file
```

**Folder Structure Evolution:**

```
Small App (< 10 components):
src/
├── components/
├── pages/
└── utils/

Medium App (10-50 components):
src/
├── features/
├── shared/
├── layouts/
└── pages/

Large App (50+ components):
src/
├── features/          # Feature modules
├── shared/            # Shared resources
├── layouts/           # Layout templates
├── pages/             # Page components
├── store/             # Global state
├── routes/            # Route configuration
├── services/          # API layer
├── types/             # TypeScript types
└── config/            # App configuration
```

**Interview Focus:**
- Explain feature-based architecture
- Justify folder structure choices
- Discuss scalability
- Show module boundaries
- Demonstrate organization patterns
