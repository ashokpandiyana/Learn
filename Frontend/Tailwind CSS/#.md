# PART 1: FOUNDATIONS

## Chapter 1: Introduction & Core Philosophy

### 1.1 What is Tailwind CSS?
- Utility-first CSS framework
- Difference from component-based frameworks (Bootstrap, Bulma)
- Atomic CSS principles

### 1.2 Why Tailwind?
- Rapid prototyping
- Consistent design systems
- Smaller production bundles (with purging)
- No context switching between HTML and CSS

### 1.3 Mental Model Shift
- **‚ö†Ô∏è IMPORTANT**: Embrace "ugly" HTML with many classes
- Composition over inheritance
- Design in the browser, not in CSS files

### 1.4 Installation Methods
- CDN (development only)
- PostCSS plugin (recommended)
- Tailwind CLI
- Framework integrations (Next.js, Vite, CRA, etc.)

> **üí° Interview Note**: Know the difference between JIT (Just-in-Time) mode vs AOT. JIT is now default since v3.0.

## Chapter 16: Dark Mode

### 16.1 Configuration Options
```javascript
// tailwind.config.js
module.exports = {
  darkMode: 'media',    // OS preference
  darkMode: 'class',    // Manual class toggle
  darkMode: 'selector', // Any selector (v3.4+)
  darkMode: ['class', '[data-theme="dark"]'], // Custom selector
}
```

### 16.2 Usage
```html
<div class="bg-white dark:bg-gray-900">
  <p class="text-black dark:text-white">...</p>
</div>
```

### 16.3 Class Strategy Implementation
```javascript
// Toggle dark mode
document.documentElement.classList.toggle('dark')
```

### 16.4 System Preference + Override
```javascript
// Check system preference
if (window.matchMedia('(prefers-color-scheme: dark)').matches) {
  document.documentElement.classList.add('dark')
}
```

> **üí° Interview Note**: Know pros/cons of each approach. 'class' gives user control but requires JS. 'media' is automatic but no user override.

```javascript
  const base = "px-4 py-2 rounded font-medium transition-colors"
  const variants = {
    primary: "bg-blue-500 text-white hover:bg-blue-600",
    secondary: "bg-gray-200 text-gray-800 hover:bg-gray-300",
  }
  return <button className={`${base} ${variants[variant]}`}>{children}</button>
}
```

### 18.2 @apply Directive
```css
/* input.css */
@layer components {
  .btn-primary {
    @apply px-4 py-2 bg-blue-500 text-white rounded hover:bg-blue-600;
  }
}
```
- **‚ö†Ô∏è WARNING**: Use sparingly. Defeats purpose of utility-first.

### 18.3 Plugin-Based Components
```javascript
// tailwind.config.js
const plugin = require('tailwindcss/plugin')

module.exports = {
  plugins: [
    plugin(function({ addComponents }) {
      addComponents({
        '.btn': {
          padding: '.5rem 1rem',
          borderRadius: '.25rem',
          fontWeight: '600',
        },
      })
    })
  ]
}
```

### 18.4 When to Use Each Approach
| Approach | Use When |
|----------|----------|
| Inline utilities | Default choice |
| Component abstraction | Framework components (React, Vue) |
| @apply | Simple, repeated patterns |
| Plugins | Shared across projects, design system |

> **üí° Interview Note**: Always advocate for component extraction over @apply for maintainability.

## Chapter 26: Common Interview Questions

### 26.1 Conceptual Questions

**Q: What makes Tailwind different from Bootstrap?**
- Utility-first vs component-based
- Customization: Config-driven vs Sass variables
- Learning curve: Steeper initially, more flexible long-term
- Bundle size: Smaller in production (purged)

**Q: How does JIT mode work?**
- Generates CSS on-demand during development
- Scans content files for class usage
- Enables arbitrary values without config
- Results in faster builds and smaller dev CSS

**Q: When would you NOT use Tailwind?**
- Email templates (limited CSS support)
- Existing projects with heavy CSS architecture
- Teams unfamiliar with utility-first
- When design varies wildly per element

### 26.2 Practical Questions

**Q: How do you handle dynamic classes?**
```javascript
// ‚ùå Won't work - classes not detected
const color = 'blue'
<div className={`bg-${color}-500`} />

// ‚úÖ Works - complete class names
const colorClasses = {
  blue: 'bg-blue-500',
  red: 'bg-red-500',
}
<div className={colorClasses[color]} />
```

**Q: How do you create a design system with Tailwind?**
1. Define colors, spacing, typography in config
2. Create component abstractions (React/Vue components)
3. Use CSS variables for dynamic theming
4. Document patterns in Storybook
5. Share config as npm package

**Q: How do you debug specificity issues?**
- Use `!important` modifier: `!bg-red-500`
- Check source order in DevTools
- Use `@layer` for proper cascade
- Consider `important: true` in config
