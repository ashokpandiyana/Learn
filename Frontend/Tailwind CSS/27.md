## Design System

### ‚úÖ Define Consistent Color Palette in Config

**Why it matters:**
- Ensures brand consistency
- Prevents arbitrary color choices
- Makes theming easier
- Reduces decision fatigue

**‚ùå Bad Practice:**

```html
<!-- Random colors chosen by feel -->
<button class="bg-[#3b7ff2] text-white">Button 1</button>
<button class="bg-[#4287f5] text-white">Button 2</button>
<button class="bg-[#2e6bd4] text-white">Button 3</button>

<!-- Problems:
  - Inconsistent blues
  - Hard to maintain
  - No semantic meaning
  - Can't theme easily
-->
```

**‚úÖ Good Practice:**

```javascript
// tailwind.config.js
module.exports = {
  theme: {
    colors: {
      // Semantic brand colors
      primary: {
        50: '#eff6ff',
        100: '#dbeafe',
        200: '#bfdbfe',
        300: '#93c5fd',
        400: '#60a5fa',
        500: '#3b82f6',  // Main brand color

        950: '#172554',

      // Neutral scale for UI
```

### ‚úÖ Establish Spacing Scale

**Why it matters:**
- Visual rhythm and consistency
- Professional appearance
- Easier responsive design
- Predictable layouts

**‚ùå Bad Practice:**

```html
<!-- Random spacing values -->
<div class="mt-[13px] mb-[27px] p-[19px]">
  <h2 class="mb-[11px]">Title</h2>
  <p class="mb-[23px]">Content</p>
</div>

<!-- Problems:
  - No consistency
  - Hard to maintain
  - Doesn't scale well
  - Looks unprofessional
-->
```

**‚úÖ Good Practice:**

**Option 1: Use Default Spacing Scale (Recommended)**

Tailwind's default scale is based on 4px increments:
```
0: 0px
1: 4px
2: 8px
3: 12px
4: 16px
6: 24px
8: 32px
12: 48px
16: 64px
```

```html
<!-- Consistent spacing from default scale -->
<div class="mt-8 mb-12 p-6">
  <h2 class="mb-4">Title</h2>
  <p class="mb-6">Content</p>
</div>
```

**Option 2: Customize for Your Brand**

```javascript
// tailwind.config.js
module.exports = {
  theme: {
    spacing: {
      // Keep defaults but add custom
      '18': '4.5rem',   // 72px
      '88': '22rem',    // 352px
      '128': '32rem',   // 512px
    },
  },
};
```

**Best Practice Guidelines:**

```
Spacing Recommendations:
‚îú‚îÄ‚îÄ Small components: 2, 3, 4 (8-16px)
‚îú‚îÄ‚îÄ Medium components: 6, 8 (24-32px)
‚îú‚îÄ‚îÄ Large sections: 12, 16, 20 (48-80px)
‚îî‚îÄ‚îÄ Page sections: 24, 32 (96-128px)

Example Application:

  <div class="mb-12">   <!-- Subsection -->
    <h2 class="mb-6">   <!-- Heading -->
      Title
    </h2>
    <p class="mb-4">    <!-- Paragraph -->
      Content
    </p>
  </div>
</section>
```

### ‚úÖ Set Up Typography Scale

**Why it matters:**
- Visual hierarchy
- Readability
- Professional appearance
- Consistent type system

**‚ùå Bad Practice:**

```html
<!-- Arbitrary font sizes -->
<h1 class="text-[37px]">Heading</h1>
<h2 class="text-[28px]">Subheading</h2>
<p class="text-[17px] leading-[29px]">Body text</p>

<!-- Problems:
  - No systematic scale
  - Inconsistent line heights
  - Hard to maintain
  - Doesn't respond well
-->
```

**‚úÖ Good Practice:**

```javascript
// tailwind.config.js
module.exports = {
  theme: {
    fontSize: {
      // Each size includes optimal line-height
      'xs': ['0.75rem', { lineHeight: '1rem' }],      // 12px / 16px
      'sm': ['0.875rem', { lineHeight: '1.25rem' }],  // 14px / 20px
      'base': ['1rem', { lineHeight: '1.5rem' }],     // 16px / 24px
      'lg': ['1.125rem', { lineHeight: '1.75rem' }],  // 18px / 28px
      'xl': ['1.25rem', { lineHeight: '1.75rem' }],   // 20px / 28px
      '2xl': ['1.5rem', { lineHeight: '2rem' }],      // 24px / 32px
      '3xl': ['1.875rem', { lineHeight: '2.25rem' }], // 30px / 36px
      '4xl': ['2.25rem', { lineHeight: '2.5rem' }],   // 36px / 40px
      '5xl': ['3rem', { lineHeight: '1.2' }],         // 48px / 57.6px
      '6xl': ['3.75rem', { lineHeight: '1.1' }],      // 60px / 66px
    },
    fontWeight: {
      thin: '100',
      light: '300',
      normal: '400',
      medium: '500',
      semibold: '600',
      bold: '700',
      black: '900',
    },
  },
};
```

**Typography Hierarchy Example:**

```html
<article class="max-w-4xl mx-auto p-8">
  <!-- H1: Display/Hero -->
  <h1 class="text-5xl font-bold text-gray-900 mb-6 leading-tight">
    Main Article Heading
  </h1>
  
  <!-- Lead paragraph -->
  <p class="text-xl text-gray-600 mb-8 leading-relaxed">
    This is the lead paragraph with slightly larger text to draw attention 
    and provide a compelling introduction to the article.
  </p>
  
  <!-- H2: Major sections -->
  <h2 class="text-3xl font-bold text-gray-900 mt-12 mb-6">
    Major Section Heading
  </h2>
  
  <!-- Body text -->
  <p class="text-base text-gray-700 mb-4 leading-relaxed">
    This is regular body text at the base font size. It should be comfortable 
    to read for extended periods.
  </p>
  
  <!-- H3: Subsections -->
  <h3 class="text-2xl font-semibold text-gray-900 mt-8 mb-4">
    Subsection Heading
  </h3>
  
  <p class="text-base text-gray-700 mb-4 leading-relaxed">
    More body content following the subsection.
  </p>
  
  <!-- Small text -->
  <p class="text-sm text-gray-600 mt-8">
    Meta information, captions, or footnotes use smaller text.
  </p>
</article>
```

**Type Scale Ratios:**

Common typographic scales:
- **1.125** (Major Second) - Subtle, elegant
- **1.200** (Minor Third) - Comfortable, standard
- **1.250** (Major Third) - Noticeable, punchy
- **1.333** (Perfect Fourth) - Strong hierarchy
- **1.414** (Augmented Fourth) - Dramatic

Tailwind's default roughly follows **1.2-1.25** ratio.

## Performance

### ‚úÖ Content Paths Are Specific (Not Overly Broad)

**Why it matters:**
- Faster build times
- Smaller CSS bundles
- Fewer false positives
- Better performance

**‚ùå Bad Practice:**

```javascript
// tailwind.config.js
module.exports = {
  content: [
    './**/*.{js,jsx,ts,tsx}',  // ‚ùå WAY too broad
    './node_modules/**/*.js',   // ‚ùå Scans dependencies (slow!)
  ],
};

// Problems:
// - Scans ALL files in project
// - Includes node_modules (thousands of files)
// - Extremely slow builds
// - May include unused utilities from libraries
```

**‚úÖ Good Practice:**

```javascript
// tailwind.config.js
module.exports = {
  content: [
    // ‚úÖ Specific to your source files
    './src/**/*.{js,jsx,ts,tsx}',
    './src/pages/**/*.{js,jsx,ts,tsx}',
    './src/components/**/*.{js,jsx,ts,tsx}',
    
    // ‚úÖ Include index.html if it has classes
    './public/index.html',
    
    // ‚úÖ Only specific third-party files if needed
    './node_modules/@company/ui-components/**/*.js',
  ],
  
  // Exclude files you know don't contain classes
  exclude: [
    './src/**/*.test.{js,jsx}',
    './src/**/*.spec.{js,jsx}',
  ],
};
```

**Framework-Specific Best Practices:**

```javascript
// Next.js
content: [
  './pages/**/*.{js,ts,jsx,tsx,mdx}',
  './components/**/*.{js,ts,jsx,tsx,mdx}',
  './app/**/*.{js,ts,jsx,tsx,mdx}',
]

// Vue
content: [
  './index.html',
  './src/**/*.{vue,js,ts,jsx,tsx}',
]

// Svelte
content: [
  './src/**/*.{html,js,svelte,ts}',
]
```

**Impact Measurement:**

```bash
# Before optimization
content: ['./**/*.{js,jsx}']
Build time: 8-12 seconds
CSS size: 45KB

# After optimization
content: ['./src/**/*.{js,jsx}']
Build time: 2-3 seconds
CSS size: 12KB
```

### ‚úÖ No Dynamic Class Name Construction

**Why it matters:**
- JIT mode can't detect dynamic classes
- Leads to missing styles in production
- Common source of bugs

**‚ùå Bad Practice:**

```jsx
// ‚ùå String interpolation - NEVER works
function Alert({ type }) {
  return (
    <div className={`bg-${type}-100 border-${type}-500`}>
      Alert
    </div>
  );
}

// ‚ùå Template literals - NEVER works
const status = 'success';
<div className={`text-${status}-600`}>

// ‚ùå Ternary with concatenation - NEVER works
<div className={isError ? `bg-red-${shade}` : `bg-green-${shade}`}>
```

**‚úÖ Good Practice:**

**Solution 1: Complete Class Names (Best)**

```jsx
function Alert({ type }) {
  const typeStyles = {
    info: 'bg-blue-100 border-blue-500 text-blue-900',
    success: 'bg-green-100 border-green-500 text-green-900',
    warning: 'bg-yellow-100 border-yellow-500 text-yellow-900',
    error: 'bg-red-100 border-red-500 text-red-900',
  };

  return (
    <div className={`p-4 border-l-4 rounded ${typeStyles[type]}`}>
      Alert content
    </div>
  );
}

// Usage
<Alert type="success" />
<Alert type="error" />
```

**Solution 2: Safelist (When Necessary)**

```javascript
// tailwind.config.js
module.exports = {
  safelist: [
    // Simple safelist
    'bg-red-500',
    'bg-green-500',
    'bg-blue-500',
    
    // Or pattern-based
    {
      pattern: /bg-(red|green|blue|yellow)-(100|500|900)/,
      variants: ['hover', 'focus'],
    },
  ],
};
```

**Solution 3: Class Variance Authority (Production)**

```jsx
import { cva } from 'class-variance-authority';

const alertVariants = cva(
  'p-4 border-l-4 rounded',
  {
    variants: {
      type: {
        info: 'bg-blue-100 border-blue-500 text-blue-900',
        success: 'bg-green-100 border-green-500 text-green-900',
        warning: 'bg-yellow-100 border-yellow-500 text-yellow-900',
        error: 'bg-red-100 border-red-500 text-red-900',
      },
    },
    defaultVariants: {
      type: 'info',
    },
  }
);

function Alert({ type, children }) {
  return (
    <div className={alertVariants({ type })}>
      {children}
    </div>
  );
}
```

### ‚úÖ Unused Utilities Are Purged

**How Purging Works:**

Tailwind's JIT mode automatically purges unused utilities by:
1. Scanning content files
2. Extracting class names
3. Generating only used utilities
4. Removing everything else

**Ensure Proper Purging:**

```javascript
// tailwind.config.js
module.exports = {
  content: [
    './src/**/*.{js,jsx,ts,tsx}', // ‚úÖ Scan source files
    './public/index.html',         // ‚úÖ Include HTML
  ],
  // No need for purge config in v3+ (automatic)
};
```

**Check Purging Effectiveness:**

```bash
# Development build
npm run dev
# CSS: ~50KB (includes all used classes)

# Production build
npm run build
# CSS: ~10KB (minified and optimized)

# If production CSS is large (>50KB), check:
# 1. Are content paths too broad?
# 2. Are you safelisting too many classes?
# 3. Are you using dynamic class names?
```

**Typical Production Sizes:**

```
Small project (landing page): 5-10KB
Medium project (e-commerce): 10-25KB
Large project (dashboard): 25-50KB

If >50KB: Something is wrong
If >100KB: Definitely investigate
```

## Code Quality

### ‚úÖ Consistent Class Ordering

**Why it matters:**
- Easier to read
- Faster to scan
- Easier to debug
- Team consistency

**Recommended Order:**

```
1. Layout Properties
   ‚îú‚îÄ‚îÄ display (flex, grid, block, etc.)
   ‚îú‚îÄ‚îÄ position (relative, absolute, fixed)
   ‚îú‚îÄ‚îÄ top/right/bottom/left
   ‚îî‚îÄ‚îÄ z-index

2. Flexbox/Grid
   ‚îú‚îÄ‚îÄ flex-direction, justify-content, align-items
   ‚îú‚îÄ‚îÄ grid-cols, grid-rows
   ‚îî‚îÄ‚îÄ gap

3. Box Model (Outside ‚Üí Inside)
   ‚îú‚îÄ‚îÄ width, height
   ‚îú‚îÄ‚îÄ margin
   ‚îú‚îÄ‚îÄ padding
   ‚îî‚îÄ‚îÄ border

4. Typography
   ‚îú‚îÄ‚îÄ font-family, font-size, font-weight
   ‚îú‚îÄ‚îÄ text-align, text-color
   ‚îî‚îÄ‚îÄ line-height, letter-spacing

5. Visual
   ‚îú‚îÄ‚îÄ background
   ‚îú‚îÄ‚îÄ border-radius, border-color
   ‚îú‚îÄ‚îÄ box-shadow
   ‚îî‚îÄ‚îÄ opacity

6. Transforms & Transitions
   ‚îú‚îÄ‚îÄ transform
   ‚îú‚îÄ‚îÄ transition
   ‚îî‚îÄ‚îÄ animation

7. Interactive States
   ‚îú‚îÄ‚îÄ hover:
   ‚îú‚îÄ‚îÄ focus:
   ‚îú‚îÄ‚îÄ active:
   ‚îî‚îÄ‚îÄ disabled:

8. Responsive Variants (smallest to largest)
   ‚îú‚îÄ‚îÄ sm:
   ‚îú‚îÄ‚îÄ md:
   ‚îú‚îÄ‚îÄ lg:
   ‚îú‚îÄ‚îÄ xl:
   ‚îî‚îÄ‚îÄ 2xl:
```

**Examples:**

```html
<!-- ‚ùå Bad: Random order -->
<div class="text-white p-4 hover:bg-blue-700 bg-blue-600 rounded-lg font-bold mt-4 flex items-center">
  Chaotic
</div>

<!-- ‚úÖ Good: Consistent order -->
<div class="flex items-center mt-4 p-4 bg-blue-600 text-white font-bold rounded-lg hover:bg-blue-700">
  Organized
</div>

<!-- ‚ùå Bad: Responsive classes scattered -->
<div class="p-4 md:p-6 text-base lg:text-xl bg-white md:bg-gray-100">
  Confusing
</div>

<!-- ‚úÖ Good: Responsive classes grouped -->
<div class="p-4 bg-white text-base md:p-6 md:bg-gray-100 lg:text-xl">
  Clear
</div>
```

**Real Component Example:**

```jsx

  return (
    // Layout ‚Üí Box Model ‚Üí Typography ‚Üí Visual ‚Üí Interactive ‚Üí Responsive
    <div className="
```

### ‚úÖ Not Overusing @apply

**Why @apply should be rare:**
- Defeats utility-first philosophy
- Loses JIT benefits
- Harder to override
- Creates abstraction layer

**‚ùå Bad Practice:**

```css

  @apply bg-white rounded-xl shadow-lg p-6 border border-gray-200;
  @apply hover:shadow-2xl transition-shadow duration-300;
  @apply mb-6 max-w-md mx-auto;
}

.card-title {
  @apply text-2xl font-bold text-gray-900 mb-4;
}

.card-body {
  @apply text-gray-600 mb-4;
}

.card-footer {
  @apply flex items-center justify-between pt-4 border-t border-gray-200;
}

/* Problems:
  - Creating CSS classes defeats utility-first purpose
  - Harder to customize per instance
  - Loses Tailwind's flexibility
  - Team must learn custom CSS classes
*/
```

```jsx

function Card({ title, children, footer }) {

</div>
```

### ‚úÖ Interactive Elements Are Keyboard Accessible

**Requirements:**
- Tab navigation works
- Enter/Space activates
- Escape closes modals/menus
- Arrow keys for lists/menus

**‚ùå Bad Practice:**

```html

</div>

<!-- Link that triggers action - wrong element -->
<a href="#" onclick="doSomething(); return false;" class="text-blue-600">
  Trigger Action
</a>
```

**‚úÖ Good Practice:**

```html
<!-- Proper button element -->
<button 
  onclick="handleClick()"
  class="px-4 py-2 bg-blue-600 text-white rounded 
         hover:bg-blue-700
         focus:outline-none focus:ring-2 focus:ring-blue-500"
>
  Click me
</button>

<!-- Button for actions -->
<button 
  onclick="doSomething()"
  class="text-blue-600 hover:text-blue-700 underline"
>
  Trigger Action
</button>

<!-- If you MUST use div, make it accessible -->

  role="button"
  tabindex="0"
  onclick="handleClick()"
  onkeypress="if(event.key==='Enter'||event.key===' ')handleClick()"

         focus:outline-none focus:ring-2 focus:ring-blue-500"
```

**Complete Keyboard Navigation Example:**

```jsx
function AccessibleMenu() {
  const [isOpen, setIsOpen] = useState(false);
  const [focusedIndex, setFocusedIndex] = useState(0);
  const items = ['Profile', 'Settings', 'Sign out'];

  const handleKeyDown = (e) => {
    switch(e.key) {
      case 'ArrowDown':
        e.preventDefault();
        setFocusedIndex((prev) => (prev + 1) % items.length);
        break;
      case 'ArrowUp':
        e.preventDefault();
        setFocusedIndex((prev) => (prev - 1 + items.length) % items.length);
        break;
      case 'Enter':
        e.preventDefault();
        handleItemClick(items[focusedIndex]);
        break;
      case 'Escape':
        setIsOpen(false);
        break;
    }
  };

  return (
    <div className="relative">
      <button
        onClick={() => setIsOpen(!isOpen)}
        onKeyDown={handleKeyDown}
        aria-expanded={isOpen}
        aria-haspopup="true"
        class="px-4 py-2 bg-blue-600 text-white rounded
               focus:outline-none focus:ring-2 focus:ring-blue-500"
      >
        Menu
      </button>

      {isOpen && (
        <ul
          role="menu"
          className="absolute top-full mt-2 w-48 bg-white rounded-lg shadow-xl border border-gray-200 py-1"
          onKeyDown={handleKeyDown}
        >
          {items.map((item, index) => (
            <li key={item} role="none">
              <button
                role="menuitem"
                className={`w-full text-left px-4 py-2 ${
                  index === focusedIndex ? 'bg-blue-500 text-white' : 'text-gray-700'
                } hover:bg-blue-500 hover:text-white transition-colors`}
                onClick={() => handleItemClick(item)}
              >
                {item}
              </button>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

### ‚úÖ Class Merging Utility for Component Libraries

**Why you need it:**

```jsx
// Without merging
<Button className="px-8">Text</Button>
// Result: "px-4 py-2 bg-blue-500 px-8"
// Problem: Both px-4 and px-8 in HTML

// With merging
<Button className="px-8">Text</Button>
// Result: "py-2 bg-blue-500 px-8"
// Better: px-4 removed automatically
```

**Setup:**

```bash
npm install tailwind-merge clsx
```

```javascript
// lib/utils.js
import { clsx } from 'clsx';
import { twMerge } from 'tailwind-merge';

export function cn(...inputs) {
  return twMerge(clsx(inputs));
}
```

**Usage in Components:**

```jsx
import { cn } from '@/lib/utils';

export function Button({ className, variant = 'primary', ...props }) {
  return (
    <button
      className={cn(
        'px-4 py-2 rounded font-semibold',
        variant === 'primary' && 'bg-blue-600 text-white',
        variant === 'secondary' && 'bg-gray-200 text-gray-900',
        className // User overrides work correctly
      )}
      {...props}
    />
  );
}

// Now users can override cleanly
<Button className="px-8 bg-purple-600">
  Custom Button
</Button>
// Result: px-8 overrides px-4, bg-purple-600 overrides bg-blue-600
```

### ‚úÖ Consistent Variant Ordering

**Standard Order:**

```
1. Responsive (smallest to largest)
   sm: ‚Üí md: ‚Üí lg: ‚Üí xl: ‚Üí 2xl:

2. State (in order of specificity)
   hover: ‚Üí focus: ‚Üí active: ‚Üí disabled:

3. Pseudo-classes
   first: ‚Üí last: ‚Üí odd: ‚Üí even:

4. Special variants
   dark: ‚Üí group-hover: ‚Üí peer-checked:
```

**‚ùå Bad Practice:**

```html
<!-- Random variant order -->
<div class="lg:text-xl hover:bg-blue-700 sm:text-base md:bg-blue-500 focus:ring-2 md:text-lg bg-blue-600 sm:bg-blue-400">
  Chaotic and hard to read
</div>
```

**‚úÖ Good Practice:**

```html
<!-- Consistent order: base ‚Üí states ‚Üí responsive -->
<div class="bg-blue-600 hover:bg-blue-700 focus:ring-2 
            sm:bg-blue-400 sm:text-base 
            md:bg-blue-500 md:text-lg 
            lg:text-xl">
  Clear and organized
</div>

<!-- Real example -->
<button class="
  px-4 py-2 bg-blue-600 text-white rounded
  hover:bg-blue-700
  focus:outline-none focus:ring-2 focus:ring-blue-500
  active:bg-blue-800
  disabled:opacity-50 disabled:cursor-not-allowed
  sm:px-6 sm:text-lg
  lg:px-8
">
  Properly Ordered Button
</button>
```
