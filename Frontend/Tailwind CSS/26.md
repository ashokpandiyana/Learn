## Conceptual Questions

### Question 1: What makes Tailwind different from Bootstrap?

**Answer Structure:**
1. State the fundamental difference
2. Provide specific examples
3. Discuss trade-offs
4. Mention when to use each

**Complete Answer:**

Tailwind CSS and Bootstrap represent two fundamentally different approaches to CSS frameworks:

**Philosophy:**
- **Tailwind**: Utility-first approach - provides low-level utility classes
- **Bootstrap**: Component-first approach - provides pre-built components

**Comparison Table:**

| Aspect | Tailwind CSS | Bootstrap |
|--------|-------------|-----------|
| **Approach** | Utility classes | Component classes |
| **HTML** | `<button class="px-4 py-2 bg-blue-500 text-white rounded">` | `<button class="btn btn-primary">` |
| **Customization** | Config file (tailwind.config.js) | Sass variables or CSS overrides |
| **Learning Curve** | Steeper initially | Easier to start |
| **Design Freedom** | High - build anything | Limited - looks like Bootstrap |
| **CSS Size (Dev)** | Small with JIT (~10KB) | Large (~150KB+) |
| **CSS Size (Prod)** | 5-10KB (purged) | 20-30KB (even with purging) |
| **JavaScript** | None (CSS only) | Requires Bootstrap JS |
| **Responsive** | Mobile-first utilities | Mobile-first components |
| **Maintainability** | Very maintainable | Can become messy |

**Code Example:**

```html
<!-- Bootstrap Approach -->
<div class="card">
  <div class="card-body">
    <h5 class="card-title">Card Title</h5>
    <p class="card-text">Card content</p>
    <a href="#" class="btn btn-primary">Button</a>
  </div>
</div>

<!-- Tailwind Approach -->
<div class="bg-white rounded-lg shadow-lg overflow-hidden">
  <div class="p-6">
    <h5 class="text-xl font-bold text-gray-900 mb-2">Card Title</h5>
    <p class="text-gray-600 mb-4">Card content</p>
    <a href="#" class="inline-block px-4 py-2 bg-blue-600 text-white rounded hover:bg-blue-700">
      Button
    </a>
  </div>
</div>
```

**Trade-offs Discussion:**

**Use Tailwind when:**
- Building custom designs
- Need complete design control
- Creating a design system
- Want smaller production bundles
- Team knows CSS well

**Use Bootstrap when:**
- Need to ship quickly
- Building internal tools
- Design doesn't matter much
- Team is less CSS-savvy
- Need pre-built JS components

**Interview Tip:** Show you understand both tools have their place. Don't just say "Tailwind is better" - explain the context where each excels.

### Question 2: Explain how JIT (Just-In-Time) mode works

**Answer Structure:**
1. Explain what JIT is
2. How it works technically
3. Benefits over AOT (Ahead-of-Time)
4. Impact on development

**Complete Answer:**

JIT (Just-In-Time) mode, default since Tailwind v3.0, generates CSS on-demand as you write your HTML.

**How It Works:**

```
1. File Watching
   ├── Tailwind watches files specified in `content` config
   ├── Monitors for changes in real-time
   └── Scans for class names

2. Class Detection
   ├── Uses regex patterns to find classes
   ├── Detects: "bg-blue-500", "hover:text-red-600", etc.
   ├── Also detects arbitrary values: "top-[117px]"
   └── Maintains a cache of found classes

3. CSS Generation
   ├── Generates CSS only for detected classes
   ├── Applies variants (hover:, md:, etc.)
   ├── No pre-generation of all possible utilities
   └── Output is minimal and fast

4. Hot Reload
   ├── Updates CSS instantly
   └── Browser refreshes with new styles
```

**Before JIT (AOT - Ahead of Time):**

```css
/* Development: Generated ALL possible utilities */
.bg-blue-50 { background-color: #eff6ff; }
.bg-blue-100 { background-color: #dbeafe; }
/* ... 1000s more utilities ... */
.hover\:bg-blue-50:hover { background-color: #eff6ff; }
/* ... ALL variants of ALL utilities ... */

/* Result: 3-4MB CSS in development! */
```

**With JIT:**

```css
/* Development: Only generates what you use */
.bg-blue-500 { background-color: #3b82f6; }
.hover\:bg-blue-700:hover { background-color: #1d4ed8; }

/* Result: 10-50KB CSS in development! */
```

**Key Benefits:**

```javascript
// 1. Arbitrary Values - Generate utilities on the fly
<div class="top-[117px] text-[#b4d4ff] w-[37.5%]">

// 2. All Variants Work - Every variant with every utility
<div class="hover:scale-105 focus:rotate-45 active:translate-x-4">

// 3. Faster Builds - Only generates what's needed
// Before: 5-10 seconds initial build
// After: 1-2 seconds initial build

// 4. Development = Production - Same CSS size
// Before: 3MB dev, 10KB production (after purge)
// After: 10KB dev, 10KB production

// 5. Better DX (Developer Experience)
// - Faster feedback loop
// - Arbitrary values for one-offs
// - No PurgeCSS configuration needed
```

**Configuration:**

```javascript
// tailwind.config.js
module.exports = {
  content: [
    './src/**/*.{js,jsx,ts,tsx}',
    './public/index.html',
  ],
  // JIT is default in v3+, no need to enable
  theme: {
    extend: {},
  },
  plugins: [],
};
```

**Common Gotchas:**

```javascript
// ❌ Dynamic class names won't work (JIT can't detect them)
const color = 'blue';
<div className={`bg-${color}-500`}> // Won't generate bg-blue-500!

// ✅ Use complete class names
const colors = {
  blue: 'bg-blue-500',
  red: 'bg-red-500'
};
<div className={colors[color]}>

// ✅ Or use safelist in config
module.exports = {
  safelist: ['bg-blue-500', 'bg-red-500']
};
```

**Interview Tip:** Emphasize that JIT eliminates the need for PurgeCSS configuration and makes development experience match production.

### Question 3: When would you NOT use Tailwind?

**Answer Structure:**
1. Show you understand limitations
2. Provide specific scenarios
3. Suggest alternatives
4. Show balanced thinking

**Complete Answer:**

While Tailwind is powerful, it's not always the best choice. Here are scenarios where you might avoid it:

**1. Email Templates**

```html
<!-- ❌ Tailwind classes don't work in email clients -->
<div class="p-4 bg-blue-500 text-white">
  Email content
</div>

<!-- ✅ Email clients require inline styles -->
<div style="padding: 16px; background-color: #3b82f6; color: #ffffff;">
  Email content
</div>
```

**Why:** Email clients have extremely limited CSS support and strip out `<style>` tags and class-based styling.

**Alternative:** Use inline styles or email-specific tools like MJML.

---

**2. Existing Large Codebases with Established CSS Architecture**

```css
/* Existing: BEM methodology with 1000+ components */
.card { }
.card__header { }
.card__header--primary { }
.card__body { }
.card__footer { }

/* Adding Tailwind creates problems:
   - Two CSS methodologies in one codebase
   - Confusion for developers
   - Larger CSS bundle (both systems loaded)
   - Inconsistent styling across codebase
   - Difficult to maintain
*/
```

**Why:** Integration overhead outweighs benefits. Migration would be massive.

**Better Approach:** If you must add it, isolate to new features only or do a complete rewrite.

---

**3. Teams Unfamiliar with Utility-First CSS**

```html
<!-- New developer sees this -->
<button class="inline-flex items-center px-4 py-2 border border-transparent text-sm font-medium rounded-md shadow-sm text-white bg-indigo-600 hover:bg-indigo-700 focus:outline-none focus:ring-2 focus:ring-offset-2 focus:ring-indigo-500">
  Click me
</button>

<!-- Common reactions:
   - "This is ugly and unreadable!"
   - "Why so many classes?"
   - "How do I find and change the background color?"
   - "This violates separation of concerns!"
*/
```

**Why:** Learning curve can significantly slow down development initially. Team resistance is real.

**When it works:** When team has time to learn and leadership buy-in exists.

**When it doesn't:** Tight deadlines, resistant team members, or contractors unfamiliar with Tailwind.

---

**4. Pixel-Perfect Print Stylesheets**

```css
/* Print requires very specific control */
@media print {
  .page-break {
    page-break-before: always;
  }
  
  .no-print {
    display: none !important;
  }
  
  /* Complex print-specific rules */
  @page {
    size: A4;
    margin: 2cm;
  }
  
  table {
    page-break-inside: avoid;
  }
}
```

**Why:** Print styling requires CSS that Tailwind doesn't cover well. `print:` variant exists but is limited.

**Better:** Traditional CSS with `@media print` and `@page` rules.

---

**5. When Every Element Needs Unique Styling**

```html
<!-- If your design looks like this -->
<div style="
  background: linear-gradient(47deg, #ff0000 0%, #00ff00 50%, #0000ff 100%);
  border: 3px dashed purple;
  border-radius: 37px;
  box-shadow: 5px 5px 0 rgba(0,0,0,0.3) inset, -3px -3px 0 rgba(255,255,255,0.3) inset;
  transform: skew(-5deg, 2deg) rotate(3deg);
">
  Unique snowflake element
</div>

<!-- Tailwind becomes awkward:
   - Arbitrary values everywhere
   - No reusability
   - Hard to maintain
   - Defeats utility-first purpose
-->
```

**Why:** Tailwind shines with systematic design. Completely unique designs for every element make utilities pointless.

**Better:** Just use regular CSS or inline styles.

---

**6. Component Libraries with Heavy CSS Dependencies**

```javascript
// Using Material-UI with Tailwind can cause conflicts
import { Button } from '@material-ui/core';

<Button className="mt-4"> // Tailwind margin
  Click me
</Button>

/* Problems:
   - CSS specificity conflicts
   - Material-UI styles override Tailwind
   - Tailwind's preflight conflicts with MUI reset
   - Bundle size increases (both libraries)
   - Inconsistent styling
*/
```

**Why:** Component libraries come with their own styling systems that conflict with Tailwind.

**Better:** Use Tailwind with headless libraries (Headless UI, Radix) or unstyled components.

---

**7. Server-Side Rendered HTML with No Build Step**

```php
<!-- PHP/Ruby/Python template with no build pipeline -->
<?php foreach($items as $item): ?>
  <div class="p-4 bg-blue-500">
    <?= $item->name ?>
  </div>
<?php endforeach; ?>

<!-- Problems:
   - No way to run Tailwind CLI
   - Can't generate classes
   - Can't use JIT mode
   - Would need to include ALL utilities (huge file)
*/
```

**Why:** Tailwind requires a build step to generate CSS.

**Better:** Use traditional CSS or a CDN build (not recommended for production).

---

**8. Government/Highly Regulated Industries**

```
Issues:
- May require audit trail for all CSS changes
- Need to prove accessibility compliance with specific tools
- May require source CSS files, not generated
- Procurement processes may not allow npm packages
- Security reviews of build tools
```

**Why:** Regulatory and compliance requirements may make Tailwind's build system problematic.

**Better:** Traditional CSS with documented authoring guidelines.

---

**Summary Table:**

| Scenario | Use Tailwind? | Alternative |
|----------|--------------|-------------|
| Email templates | ❌ Never | Inline styles, MJML |
| Large existing codebase | ❌ Usually no | Continue existing approach |
| Print-heavy sites | ❌ Probably not | Traditional CSS with @media print |
| Unique designs everywhere | ❌ No | Regular CSS, styled-components |
| Teams unfamiliar with it | ⚠️ Maybe | Traditional CSS until trained |
| With MUI/Bootstrap | ❌ No | Use library's system or choose one |
| New greenfield projects | ✅ Yes | Perfect use case |
| Design systems | ✅ Yes | Tailwind excels here |
| With headless UI libs | ✅ Yes | Made for each other |

**Interview Tip:** This question tests if you're a "Tailwind zealot" or a pragmatic engineer. Show you can assess trade-offs and choose the right tool for the job.

## Practical Questions

### Question 1: How do you handle dynamic classes?

**The Problem:**

This is one of the most common Tailwind mistakes and a frequent interview topic.

```jsx
// ❌ WRONG - This won't work!
function Button({ color }) {
  return (
    <button className={`bg-${color}-500 hover:bg-${color}-700`}>
      Click me
    </button>
  );
}

// Usage
<Button color="blue" />  // bg-blue-500 won't be generated!
<Button color="red" />   // bg-red-500 won't be generated!
```

**Why It Fails:**

```
Tailwind's JIT scans for complete class names:
✅ Detects: "bg-blue-500"
❌ Doesn't detect: "bg-" + variable + "-500"

The scanner uses regex patterns:
/\bbg-blue-500\b/  ✅ Matches
/\bbg-\$\{color\}-500\b/  ❌ Doesn't match
```

**Solution 1: Complete Class Names (Recommended)**

```jsx
function Button({ variant }) {
  const variants = {
    primary: 'bg-blue-500 hover:bg-blue-700 text-white',
    secondary: 'bg-gray-200 hover:bg-gray-300 text-gray-900',
    danger: 'bg-red-500 hover:bg-red-700 text-white',
    success: 'bg-green-500 hover:bg-green-700 text-white',
  };

  return (
    <button className={`px-4 py-2 rounded font-semibold ${variants[variant]}`}>
      Click me
    </button>
  );
}

// Usage
<Button variant="primary" />
<Button variant="danger" />
```

**Why This Works:**
- All class names are complete strings
- Tailwind scanner detects them all
- Clear and maintainable
- Easy to extend

---

**Solution 2: Safelist in Config**

```javascript
// tailwind.config.js
module.exports = {
  content: ['./src/**/*.{js,jsx,ts,tsx}'],
  safelist: [
    'bg-blue-500',
    'bg-red-500',
    'bg-green-500',
    'hover:bg-blue-700',
    'hover:bg-red-700',
    'hover:bg-green-700',
  ],
  theme: {},
  plugins: [],
};
```

```jsx
// Now this works (but not recommended)
function Button({ color }) {
  return (
    <button className={`bg-${color}-500 hover:bg-${color}-700`}>
      Click me
    </button>
  );
}
```

**Why This Works:**
- Forces Tailwind to include these classes
- Works with dynamic concatenation

**Downsides:**
- Manual maintenance
- Easy to forget classes
- Can bloat bundle if overused

---

**Solution 3: Safelist with Patterns (Better for Many Colors)**

```javascript
// tailwind.config.js
module.exports = {
  safelist: [
    {
      pattern: /bg-(red|green|blue|yellow|purple)-(400|500|600)/,
      variants: ['hover', 'focus', 'active'],
    },
    {
      pattern: /text-(red|green|blue|yellow|purple)-(400|500|600)/,
      variants: ['hover'],
    },
  ],
};
```

```jsx
// Now many combinations work
function Alert({ color, text }) {
  return (
    <div className={`bg-${color}-100 border-l-4 border-${color}-500 p-4`}>
      <p className={`text-${color}-700`}>{text}</p>
    </div>
  );
}

// Usage
<Alert color="blue" text="Information" />
<Alert color="red" text="Error" />
<Alert color="green" text="Success" />
```

**When to Use:**
- Many color variations needed
- Utilities follow a pattern
- Still maintain some control

---

**Solution 4: Class Variance Authority (Advanced)**

```bash
npm install class-variance-authority
```

```jsx
import { cva } from 'class-variance-authority';

const buttonVariants = cva(
  // Base styles
  'px-4 py-2 rounded font-semibold transition-colors focus:outline-none focus:ring-2',
  {
    variants: {
      intent: {
        primary: 'bg-blue-500 text-white hover:bg-blue-700 focus:ring-blue-300',
        secondary: 'bg-gray-200 text-gray-900 hover:bg-gray-300 focus:ring-gray-400',
        danger: 'bg-red-500 text-white hover:bg-red-700 focus:ring-red-300',
      },
      size: {
        small: 'text-sm px-3 py-1.5',
        medium: 'text-base px-4 py-2',
        large: 'text-lg px-6 py-3',
      },
      outlined: {
        true: 'border-2 bg-transparent',
        false: '',
      },
    },
    compoundVariants: [
      {
        intent: 'primary',
        outlined: true,
        class: 'border-blue-500 text-blue-500 hover:bg-blue-50',
      },
    ],
    defaultVariants: {
      intent: 'primary',
      size: 'medium',
      outlined: false,
    },
  }
);

function Button({ intent, size, outlined, className, children, ...props }) {
  return (
    <button 
      className={buttonVariants({ intent, size, outlined, className })}
      {...props}
    >
      {children}
    </button>
  );
}

// Usage
<Button>Default</Button>
<Button intent="danger" size="large">Large Danger</Button>
<Button intent="primary" outlined>Outlined Primary</Button>
```

**Why This is Better:**
- Type-safe (with TypeScript)
- Compound variants (combinations)
- Clear API
- Maintainable
- Scales well

### Challenge 2: Create a Form with Validation

**Requirements:**
- Email, password, confirm password fields
- Real-time validation
- Error messages
- Success states
- Accessible
- Submit button with loading state

**Complete Solution:**

```jsx
import { useState } from 'react';

function SignUpForm() {
  const [formData, setFormData] = useState({
    email: '',
    password: '',
    confirmPassword: '',
  });

  const [errors, setErrors] = useState({});
  const [touched, setTouched] = useState({});
  const [isSubmitting, setIsSubmitting] = useState(false);
  const [submitSuccess, setSubmitSuccess] = useState(false);

  // Validation functions
  const validateEmail = (email) => {
    if (!email) return 'Email is required';
    if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
      return 'Invalid email address';
    }
    return '';
  };

  const validatePassword = (password) => {
    if (!password) return 'Password is required';
    if (password.length < 8) return 'Password must be at least 8 characters';
    if (!/(?=.*[a-z])/.test(password)) return 'Password must contain lowercase letter';
    if (!/(?=.*[A-Z])/.test(password)) return 'Password must contain uppercase letter';
    if (!/(?=.*\d)/.test(password)) return 'Password must contain number';
    return '';
  };

  const validateConfirmPassword = (confirmPassword) => {
    if (!confirmPassword) return 'Please confirm your password';
    if (confirmPassword !== formData.password) return 'Passwords do not match';
    return '';
  };

  const calculatePasswordStrength = (password) => {
    let strength = 0;
    if (password.length >= 8) strength++;
    if (/[a-z]/.test(password) && /[A-Z]/.test(password)) strength++;
    if (/\d/.test(password)) strength++;
    if (/[^a-zA-Z\d]/.test(password)) strength++;
    return strength;
  };

  // Event handlers
  const handleChange = (e) => {
    const { name, value } = e.target;
    setFormData(prev => ({ ...prev, [name]: value }));

    // Validate on change if field was touched
    if (touched[name]) {
      const validator = {
        email: validateEmail,
        password: validatePassword,
        confirmPassword: validateConfirmPassword,
      }[name];
      
      setErrors(prev => ({
        ...prev,
        [name]: validator(value)
      }));
    }
  };

  const handleBlur = (e) => {
    const { name, value } = e.target;
    setTouched(prev => ({ ...prev, [name]: true }));

    const validator = {
      email: validateEmail,
      password: validatePassword,
      confirmPassword: validateConfirmPassword,
    }[name];
    
    setErrors(prev => ({
      ...prev,
      [name]: validator(value)
    }));
  };

  const handleSubmit = async (e) => {
    e.preventDefault();

    // Validate all fields
    const newErrors = {
      email: validateEmail(formData.email),
      password: validatePassword(formData.password),
      confirmPassword: validateConfirmPassword(formData.confirmPassword),
    };

    setErrors(newErrors);
    setTouched({
      email: true,
      password: true,
      confirmPassword: true,
    });

    // Check if any errors
    if (Object.values(newErrors).some(error => error)) {
      return;
    }

    // Submit form
    setIsSubmitting(true);
    try {
      // Simulate API call
      await new Promise(resolve => setTimeout(resolve, 2000));
      setSubmitSuccess(true);
      console.log('Form submitted:', formData);
    } catch (error) {
      console.error('Submit error:', error);
    } finally {
      setIsSubmitting(false);
    }
  };

  const passwordStrength = calculatePasswordStrength(formData.password);
  const strengthLabel = ['Weak', 'Fair', 'Good', 'Strong'][passwordStrength];
  const strengthColor = ['red', 'yellow', 'blue', 'green'][passwordStrength];

  if (submitSuccess) {
    return (
      <div className="max-w-md mx-auto p-8 bg-white rounded-xl shadow-lg">
        <div className="text-center">
          <div className="w-16 h-16 bg-green-100 rounded-full flex items-center justify-center mx-auto mb-4">
            <svg className="w-10 h-10 text-green-600" fill="none" stroke="currentColor" viewBox="0 0 24 24">
              <path strokeLinecap="round" strokeLinejoin="round" strokeWidth="2" d="M5 13l4 4L19 7" />
            </svg>
          </div>
          <h2 className="text-2xl font-bold text-gray-900 mb-2">Success!</h2>
          <p className="text-gray-600">Your account has been created.</p>
        </div>
      </div>
    );
  }

  return (
    <form onSubmit={handleSubmit} className="max-w-md mx-auto p-8 bg-white rounded-xl shadow-lg">
      <h2 className="text-3xl font-bold text-gray-900 mb-8">Create Account</h2>

      {/* Email */}
      <div className="mb-6">
        <label htmlFor="email" className="block text-sm font-medium text-gray-700 mb-2">
          Email Address *
        </label>
        <input
          type="email"
          id="email"
          name="email"
          value={formData.email}
          onChange={handleChange}
          onBlur={handleBlur}
          className={`w-full px-4 py-3 border-2 rounded-lg transition-all
            ${errors.email && touched.email
              ? 'border-red-500 bg-red-50 focus:ring-red-500'
              : 'border-gray-300 focus:border-blue-500 focus:ring-blue-500'
            }
            focus:outline-none focus:ring-2`}
          placeholder="you@example.com"
          aria-invalid={errors.email && touched.email ? 'true' : 'false'}
          aria-describedby={errors.email && touched.email ? 'email-error' : undefined}
        />
        {errors.email && touched.email && (
          <p id="email-error" className="mt-2 text-sm text-red-600 flex items-center gap-1">
            <svg className="w-4 h-4" fill="currentColor" viewBox="0 0 20 20">
              <path fillRule="evenodd" d="M18 10a8 8 0 11-16 0 8 8 0 0116 0zm-7 4a1 1 0 11-2 0 1 1 0 012 0zm-1-9a1 1 0 00-1 1v4a1 1 0 102 0V6a1 1 0 00-1-1z" clipRule="evenodd" />
            </svg>
            {errors.email}
          </p>
        )}
      </div>

      {/* Password */}
      <div className="mb-6">
        <label htmlFor="password" className="block text-sm font-medium text-gray-700 mb-2">
          Password *
        </label>
        <input
          type="password"
          id="password"
          name="password"
          value={formData.password}
          onChange={handleChange}
          onBlur={handleBlur}
          className={`w-full px-4 py-3 border-2 rounded-lg transition-all
            ${errors.password && touched.password
              ? 'border-red-500 bg-red-50 focus:ring-red-500'
              : 'border-gray-300 focus:border-blue-500 focus:ring-blue-500'
            }
            focus:outline-none focus:ring-2`}
          placeholder="Min 8 characters"
          aria-invalid={errors.password && touched.password ? 'true' : 'false'}
          aria-describedby={errors.password && touched.password ? 'password-error' : undefined}
        />
        
        {/* Password strength meter */}
        {formData.password && !errors.password && (
          <div className="mt-2">
            <div className="flex items-center justify-between mb-1">
              <span className="text-xs text-gray-600">Password strength:</span>
              <span className={`text-xs font-semibold text-${strengthColor}-600`}>
                {strengthLabel}
              </span>
            </div>
            <div className="h-2 bg-gray-200 rounded-full overflow-hidden">
              <div
                className={`h-full bg-${strengthColor}-500 transition-all duration-300`}
                style={{ width: `${(passwordStrength / 4) * 100}%` }}
              />
            </div>
          </div>
        )}
        
        {errors.password && touched.password && (
          <p id="password-error" className="mt-2 text-sm text-red-600 flex items-center gap-1">
            <svg className="w-4 h-4" fill="currentColor" viewBox="0 0 20 20">
              <path fillRule="evenodd" d="M18 10a8 8 0 11-16 0 8 8 0 0116 0zm-7 4a1 1 0 11-2 0 1 1 0 012 0zm-1-9a1 1 0 00-1 1v4a1 1 0 102 0V6a1 1 0 00-1-1z" clipRule="evenodd" />
            </svg>
            {errors.password}
          </p>
        )}
      </div>

      {/* Confirm Password */}
      <div className="mb-6">
        <label htmlFor="confirmPassword" className="block text-sm font-medium text-gray-700 mb-2">
          Confirm Password *
        </label>
        <input
          type="password"
          id="confirmPassword"
          name="confirmPassword"
          value={formData.confirmPassword}
          onChange={handleChange}
          onBlur={handleBlur}
          className={`w-full px-4 py-3 border-2 rounded-lg transition-all
            ${errors.confirmPassword && touched.confirmPassword
              ? 'border-red-500 bg-red-50 focus:ring-red-500'
              : formData.confirmPassword && !errors.confirmPassword
              ? 'border-green-500 bg-green-50 focus:ring-green-500'
              : 'border-gray-300 focus:border-blue-500 focus:ring-blue-500'
            }
            focus:outline-none focus:ring-2`}
          placeholder="Re-enter password"
          aria-invalid={errors.confirmPassword && touched.confirmPassword ? 'true' : 'false'}
          aria-describedby={errors.confirmPassword && touched.confirmPassword ? 'confirm-password-error' : undefined}
        />
        {errors.confirmPassword && touched.confirmPassword && (
          <p id="confirm-password-error" className="mt-2 text-sm text-red-600 flex items-center gap-1">
            <svg className="w-4 h-4" fill="currentColor" viewBox="0 0 20 20">
              <path fillRule="evenodd" d="M18 10a8 8 0 11-16 0 8 8 0 0116 0zm-7 4a1 1 0 11-2 0 1 1 0 012 0zm-1-9a1 1 0 00-1 1v4a1 1 0 102 0V6a1 1 0 00-1-1z" clipRule="evenodd" />
            </svg>
            {errors.confirmPassword}
          </p>
        )}
        {formData.confirmPassword && !errors.confirmPassword && touched.confirmPassword && (
          <p className="mt-2 text-sm text-green-600 flex items-center gap-1">
            <svg className="w-4 h-4" fill="currentColor" viewBox="0 0 20 20">
              <path fillRule="evenodd" d="M10 18a8 8 0 100-16 8 8 0 000 16zm3.707-9.293a1 1 0 00-1.414-1.414L9 10.586 7.707 9.293a1 1 0 00-1.414 1.414l2 2a1 1 0 001.414 0l4-4z" clipRule="evenodd" />
            </svg>
            Passwords match!
          </p>
        )}
      </div>

      {/* Submit Button */}
      <button
        type="submit"
        disabled={isSubmitting}
        className="w-full py-3 bg-blue-600 text-white font-semibold rounded-lg
                   hover:bg-blue-700 active:bg-blue-800
                   disabled:opacity-50 disabled:cursor-not-allowed
                   transition-all duration-200
                   focus:outline-none focus:ring-4 focus:ring-blue-300"
      >
        {isSubmitting ? (
          <span className="flex items-center justify-center gap-2">
            <svg className="animate-spin h-5 w-5" fill="none" viewBox="0 0 24 24">
              <circle className="opacity-25" cx="12" cy="12" r="10" stroke="currentColor" strokeWidth="4" />
              <path className="opacity-75" fill="currentColor" d="M4 12a8 8 0 018-8V0C5.373 0 0 5.373 0 12h4zm2 5.291A7.962 7.962 0 014 12H0c0 3.042 1.135 5.824 3 7.938l3-2.647z" />
            </svg>
            Creating Account...
          </span>
        ) : (
          'Create Account'
        )}
      </button>

      <p className="mt-4 text-center text-sm text-gray-600">
        Already have an account?{' '}
        <a href="/login" className="text-blue-600 hover:underline font-medium">
          Sign in
        </a>
      </p>
    </form>
  );
}

export default SignUpForm;
```

**Key Points:**

1. **Real-time Validation**: On blur and on change after first blur
2. **Visual Feedback**: Different colors for error, success, normal
3. **Password Strength**: Visual indicator with dynamic width
4. **Accessibility**: ARIA attributes, proper labels, error associations
5. **Loading State**: Disabled button with spinner
6. **Success Screen**: Positive feedback after submission
