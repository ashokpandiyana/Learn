Q
What is the difference between REST and GraphQL?
‚úì IDEAL ANSWER

GraphQL and REST are both API paradigms, but they differ fundamentally in how clients request data:

Aspect REST GraphQL
Endpoints Multiple (one per resource) Single endpoint
Data Fetching Fixed response shape, often over-fetching Client specifies exactly what they need
Type System No built-in types Strongly typed schema
Versioning URL versioning (/v1/, /v2/) Schema evolution, no versioning needed
Documentation External (Swagger, etc.) Built-in introspection

üí° Interview Tip

Mention the N+1 problem: "With REST, fetching a user's posts and each post's comments requires multiple requests. GraphQL solves this with a single query that specifies the nested data structure."

Q
Explain Apollo Client's caching mechanism
‚úì IDEAL ANSWER

Apollo Client uses a normalized cache that stores data as a flat collection of objects:

// How Apollo stores data internally { "ROOT_QUERY": { "users": [ { "__ref": "User:1" }, { "__ref": "User:2" } ] }, "User:1": { "__typename": "User", "id": "1", "name": "John" }, "User:2": { "__typename": "User", "id": "2", "name": "Jane" } }
Normalization
Objects stored by __typename:id
Auto Updates
Change User:1 ‚Üí all components update
Deduplication
Same object never stored twice
References
Objects link via __ref pointers
Q
What are the different fetch policies and when would you use each?
‚úì IDEAL ANSWER
Policy Behavior Use Case
cache-first Read cache ‚Üí fetch if miss Default. Static data like user profiles
cache-and-network Show cache immediately, then fetch Dashboards, feeds - instant UX + fresh data
network-only Always fetch, update cache After mutations, critical fresh data
cache-only Cache only, never fetch Offline mode, reading already-fetched data
no-cache Always fetch, don't cache Highly sensitive data, one-time lookups
// Example: cache-and-network for a dashboard const { data, loading } = useQuery(GET_DASHBOARD, { fetchPolicy: 'cache-and-network', // Shows cached data instantly, fetches updates in background });
Q
How do you handle authentication in Apollo Client?
‚úì IDEAL ANSWER

Authentication is handled through the Apollo Link chain, specifically using an auth link that injects tokens into request headers:

import { setContext } from '@apollo/client/link/context'; const authLink = setContext((_, { headers }) => { const token = localStorage.getItem('token'); return { headers: { ...headers, authorization: token ? `Bearer ${token}` : '', }, }; }); // Link chain order: authLink ‚Üí errorLink ‚Üí httpLink const client = new ApolloClient({ link: from([authLink, errorLink, httpLink]), cache: new InMemoryCache(), });

For token refresh: Use an error link to catch UNAUTHENTICATED errors, refresh the token, and retry the request using fromPromise and forward(operation).

Optimistic UI shows the expected result immediately while the mutation runs in the background. This creates a snappy, responsive user experience.

üìù Design a Real-time Collaborative Document Editor

Key considerations:

  • Subscriptions: Use WebSocket subscriptions for real-time cursor positions and text changes
  • Optimistic UI: Show local changes immediately while syncing
  • Conflict Resolution: Implement Operational Transforms (OT) or CRDTs for concurrent edits
  • Pagination: Cursor-based pagination for document revision history
  • Caching: Cache document fragments, use cache-and-network for auto-save

üìä Optimize a Slow Dashboard Loading Many Queries

Optimization strategies:

  • Batch queries: Use BatchHttpLink to combine multiple operations
  • Fragments: Share common fields across queries
  • @defer directive: Load critical data first, defer expensive fields
  • Prefetching: Load data on hover or route transition
  • Fetch policy: cache-and-network for instant perceived load
  • Persisted queries: Reduce payload size with query hashing
