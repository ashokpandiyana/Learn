12.1 Remote Debugging

Debug websites on real mobile devices or remote browsers using Chrome's remote debugging feature.

Android Remote Debugging

  1. On Android Device:
    • Enable Developer Options (tap Build Number 7 times)

  2. On Desktop Chrome:

    • Your device and open tabs will appear
    • Click "Inspect" next to any tab

12.8 Top Interview Questions

Fundamentals

Q1: How do you debug JavaScript errors in production? Medium
Answer:
  1. Check Console for error messages and stack traces
  2. Use source maps to see original code (not minified)
  3. Set breakpoints in Sources panel at error location
  4. Use conditional breakpoints to catch specific conditions
  5. Check Network panel for failed requests
  6. Use try-catch blocks with detailed error logging
  7. Implement error monitoring (Sentry, LogRocket, etc.)
Q2: What's the difference between $0 and $()? Easy
Answer: $0 refers to the currently selected element in the Elements panel ($1-$4 are previously selected). $(selector) is an alias for document.querySelector() and $$(selector) is for querySelectorAll(). Note: $(selector) only works in Console, not in page JavaScript!
Q3: How do you find and fix memory leaks? Hard
Answer:
  1. Use Performance Monitor to watch JS heap size over time
  2. Take heap snapshots before and after suspected leak
  3. Use Three Snapshot Technique: baseline ‚Üí action ‚Üí undo ‚Üí compare
  4. Look for "Detached" DOM nodes in heap snapshot
  5. Use Retainers panel to trace why objects aren't garbage collected
  6. Common causes: forgotten event listeners, timers, closures, global variables
  7. Fix by removing listeners, clearing intervals, nullifying references

Performance

Q4: How would you improve a website's Core Web Vitals? Hard
Answer:
  • LCP (< 2.5s): Optimize server response, preload LCP image, eliminate render-blocking resources, use CDN, optimize images (WebP)
  • INP (< 200ms): Break up long tasks, debounce handlers, use Web Workers for heavy computation, minimize main thread work
  • CLS (< 0.1): Set dimensions on images/videos, reserve space for ads, use font-display: swap, avoid inserting content above existing content
Q5: What causes layout thrashing and how do you fix it? Hard
Answer: Layout thrashing occurs when you read layout properties (offsetWidth, getBoundingClientRect) and then write styles in a loop, forcing the browser to recalculate layout repeatedly. 
// BAD - Forces layout on each iteration
elements.forEach(el => {
    const width = el.offsetWidth;  // Read ‚Üí triggers layout
    el.style.width = width + 10 + 'px';  // Write
});

// GOOD - Batch reads, then batch writes
const widths = elements.map(el => el.offsetWidth);  // All reads
elements.forEach((el, i) => {
    el.style.width = widths[i] + 10 + 'px';  // All writes
});

Debugging

Q6: When would you use a conditional breakpoint vs a logpoint? Medium
Answer:
  • Conditional breakpoint: Pause only when condition is true. Use when you want to stop at a specific case (e.g., userId === 'problematic_user')
  • Logpoint: Log values without pausing. Use when you want to trace execution flow without interrupting it. Better than adding console.log to code.
Q7: How do you debug asynchronous code? Medium
Answer:
  1. Enable "Async" checkbox in Call Stack to see full async chain
  2. Use async stack traces to see Promise chain
  3. Set breakpoints in .then() handlers or after await
  4. Use XHR/Fetch breakpoints for network requests
  5. Step through with F9 (step) which respects async boundaries
  6. Add Event Listener breakpoints for setTimeout/setInterval

Network & Security

Q8: How do you debug CORS errors? Medium
Answer:
  1. Check Console for specific CORS error message
  2. Look at Network panel for the blocked request
  3. Check if there's a preflight OPTIONS request
  4. Verify server sends correct headers: Access-Control-Allow-Origin, Methods, Headers
  5. For credentialed requests, ensure Access-Control-Allow-Credentials: true
  6. Remember: Origin header must be exact (not wildcard) for credentials
Q9: What does TTFB indicate and how do you improve it? Medium
Answer: TTFB (Time to First Byte) measures time from request start until first byte of response. High TTFB indicates server-side issues.
  • Server processing time (optimize backend code, database queries)
  • Network latency (use CDN for geographic distribution)
  • DNS lookup (use faster DNS, enable DNS prefetch)
  • TLS handshake (upgrade to TLS 1.3, enable session resumption)
Q10: How do you find unused CSS and JavaScript? Easy
Answer: Use the Coverage panel (Ctrl+Shift+P ‚Üí "Show Coverage"). Click reload to start recording, then interact with the page. Results show percentage of used vs unused code for each file. Red lines indicate unused code. Solutions: code splitting, tree shaking, lazy loading, PurgeCSS for unused styles.

Before the Interview

  • Know 5+ Command Menu commands by heart
  • Be ready to demonstrate debugging live
  • Understand Core Web Vitals deeply
  • Practice explaining concepts simply

During the Interview

  • Think out loud - show your debugging process
  • Start with simple tools before advanced ones
  • Mention trade-offs (e.g., Performance vs Insights panel)
  • Connect DevTools to real production scenarios
  • Ask clarifying questions about the problem

  • Performance optimization (LCP, CLS, TBT/INP)
  • Network analysis (TTFB, Waterfall, CORS)
  • Security (CSP, Mixed content, SameSite cookies)
  • Debugging techniques (Breakpoints, Call stack, Async)
The best DevTools users don't just know the features - they know WHEN to use each tool. Practice identifying problems first, then selecting the right panel!
