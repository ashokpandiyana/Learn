18.1 tsconfig.json Deep Dive

üìã What is tsconfig.json?

The tsconfig.json file is the configuration file for TypeScript compiler. It specifies compiler options and which files to include/exclude. A properly configured tsconfig is crucial for type safety, performance, and developer experience.

üèóÔ∏è Complete Production tsconfig.json

{
  "compilerOptions": {
    /* Language and Environment */
    "target": "ES2022",                    // Modern JavaScript features
    "lib": ["ES2022", "DOM", "DOM.Iterable"], // Available APIs
    "jsx": "react-jsx",                    // JSX transformation (React 17+)
    "experimentalDecorators": true,        // Enable decorators
    "emitDecoratorMetadata": true,         // Metadata for decorators

    /* Modules */
    "module": "ESNext",                    // Use ES modules
    "moduleResolution": "bundler",         // For Vite/modern bundlers
    "resolveJsonModule": true,             // Import JSON files
    "allowImportingTsExtensions": true,    // Import .ts/.tsx files
    
    /* Emit */
    "declaration": true,                   // Generate .d.ts files
    "declarationMap": true,                // Source maps for .d.ts
    "sourceMap": true,                     // Generate source maps
    "outDir": "./dist",                    // Output directory
    "removeComments": true,                // Strip comments
    "noEmit": true,                        // Don't emit (let Vite handle)
    
    /* Interop Constraints */
    "isolatedModules": true,               // Each file is a module
    "esModuleInterop": true,               // Nicer import syntax
    "allowSyntheticDefaultImports": true,  // Allow default imports
    "forceConsistentCasingInFileNames": true, // Case-sensitive imports
    
    /* Type Checking - STRICT MODE */
    "strict": true,                        // Enable ALL strict checks
    "noImplicitAny": true,                 // No implicit 'any'
    "strictNullChecks": true,              // Null safety
    "strictFunctionTypes": true,           // Strict function checking
    "strictBindCallApply": true,           // Type-safe bind/call/apply
    "strictPropertyInitialization": true,  // Class properties must be initialized
    "noImplicitThis": true,                // 'this' must have type
    "alwaysStrict": true,                  // Use 'use strict'
    
    /* Additional Checks */
    "noUnusedLocals": true,                // Error on unused variables
    "noUnusedParameters": true,            // Error on unused parameters
    "noImplicitReturns": true,             // All paths must return
    "noFallthroughCasesInSwitch": true,    // No fallthrough in switch
    "noUncheckedIndexedAccess": true,      // Array/object access returns T | undefined
    "exactOptionalPropertyTypes": true,    // Distinguish undefined vs missing
    "noPropertyAccessFromIndexSignature": true, // Use [] for index signatures
    
    /* Completeness */
    "skipLibCheck": true,                  // Skip .d.ts file checking (faster)
    
    /* Path Mapping */
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"],
      "@components/*": ["./src/components/*"],
      "@utils/*": ["./src/utils/*"],
      "@hooks/*": ["./src/hooks/*"],
      "@types/*": ["./src/types/*"]
    }
  },
  
  /* What to include/exclude */
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist", "**/*.spec.ts"]
}

üéØ Compiler Options Explained

üéØ Target & Module

target: JavaScript version to output

module: Module system to use

Use ES2022 for modern browsers, ESNext for bundlers

üîç Module Resolution

moduleResolution: "bundler" for Vite/Webpack

moduleResolution: "node" for Node.js

Controls how imports are resolved

‚ö° JSX Configuration

jsx: "react-jsx" - React 17+ (no import React)

jsx: "react" - Classic React (need import)

jsx: "preserve" - Keep JSX as-is

üõ°Ô∏è Strict Mode

strict: true enables:

  • noImplicitAny
  • strictNullChecks
  • strictFunctionTypes
  • strictBindCallApply
  • strictPropertyInitialization
  • noImplicitThis
  • alwaysStrict

‚ö†Ô∏è Critical Options for Safety

Option What It Does Why Important
noImplicitAny Errors on implicit 'any' types Prevents accidental loss of type safety
strictNullChecks null/undefined not assignable to other types Prevents null reference errors
noUncheckedIndexedAccess Array/object access includes undefined Safer array/object access
exactOptionalPropertyTypes Distinguishes undefined vs missing prop More accurate optional property handling
noImplicitReturns All code paths must return a value Prevents missing return statements
‚ö†Ô∏è Interview Alert: Be ready to explain what strict: true enables and why each option matters! This is a common interview question.

üó∫Ô∏è Path Mapping

Path mapping allows you to create clean import aliases instead of relative paths:

// Without path mapping ‚ùå
import { Button } from '../../../components/Button';
import { useAuth } from '../../../../hooks/useAuth';

// With path mapping ‚úÖ
import { Button } from '@components/Button';
import { useAuth } from '@hooks/useAuth';

// Configuration
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"],
      "@components/*": ["./src/components/*"],
      "@hooks/*": ["./src/hooks/*"],
      "@utils/*": ["./src/utils/*"],
      "@types/*": ["./src/types/*"],
      "@api/*": ["./src/api/*"]
    }
  }
}
üí° Vite Configuration: You also need to configure Vite to recognize these paths: 
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
      '@components': path.resolve(__dirname, './src/components'),
      '@hooks': path.resolve(__dirname, './src/hooks'),
      '@utils': path.resolve(__dirname, './src/utils'),
      '@types': path.resolve(__dirname, './src/types'),
      '@api': path.resolve(__dirname, './src/api'),
    },
  },
});

üì¶ Project References

For monorepos, use project references to improve build performance:

// packages/shared/tsconfig.json
{
  "compilerOptions": {
    "composite": true,  // Enable project references
    "declaration": true,
    "outDir": "./dist"
  },
  "include": ["src"]
}

// packages/app/tsconfig.json
{
  "compilerOptions": {
    "outDir": "./dist"
  },
  "references": [
    { "path": "../shared" }  // Reference shared package
  ]
}

// Root tsconfig.json
{
  "files": [],
  "references": [
    { "path": "./packages/shared" },
    { "path": "./packages/app" }
  ]
}

18.2 ESLint Configuration

üîç Why ESLint with TypeScript?

While TypeScript catches type errors, ESLint catches code quality issues, potential bugs, and enforces coding standards. The @typescript-eslint plugin adds TypeScript-specific linting rules.

üì¶ Installation

npm install -D eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin
npm install -D eslint-plugin-react eslint-plugin-react-hooks
npm install -D eslint-config-prettier  # Disable formatting rules

‚öôÔ∏è Complete ESLint Configuration

// .eslintrc.cjs
module.exports = {
  root: true,
  env: {
    browser: true,
    es2022: true,
    node: true,
  },
  
  extends: [
    'eslint:recommended',
    'plugin:@typescript-eslint/recommended',
    'plugin:@typescript-eslint/recommended-requiring-type-checking',
    'plugin:react/recommended',
    'plugin:react-hooks/recommended',
    'plugin:react/jsx-runtime',  // No need to import React
    'prettier',  // Must be last to override others
  ],
  
  parser: '@typescript-eslint/parser',
  parserOptions: {
    ecmaVersion: 'latest',
    sourceType: 'module',
    project: ['./tsconfig.json'],  // Required for type-aware rules
    tsconfigRootDir: __dirname,
    ecmaFeatures: {
      jsx: true,
    },
  },
  
  plugins: [
    '@typescript-eslint',
    'react',
    'react-hooks',
  ],
  
  settings: {
    react: {
      version: 'detect',  // Auto-detect React version
    },
  },
  
  rules: {
    /* TypeScript-specific rules */
    '@typescript-eslint/no-explicit-any': 'error',  // Ban 'any'
    '@typescript-eslint/explicit-function-return-type': 'off',  // Let inference work
    '@typescript-eslint/explicit-module-boundary-types': 'off',
    '@typescript-eslint/no-unused-vars': ['error', { 
      argsIgnorePattern: '^_',  // Allow _unused variables
      varsIgnorePattern: '^_',
    }],
    '@typescript-eslint/no-floating-promises': 'error',  // Must handle promises
    '@typescript-eslint/no-misused-promises': 'error',
    '@typescript-eslint/await-thenable': 'error',  // Only await promises
    '@typescript-eslint/no-unnecessary-type-assertion': 'error',
    '@typescript-eslint/strict-boolean-expressions': 'warn',
    '@typescript-eslint/prefer-nullish-coalescing': 'warn',
    '@typescript-eslint/prefer-optional-chain': 'warn',
    '@typescript-eslint/consistent-type-imports': ['warn', {
      prefer: 'type-imports',  // Use "import type"
    }],
    '@typescript-eslint/consistent-type-definitions': ['error', 'interface'],
    
    /* React rules */
    'react/prop-types': 'off',  // Not needed with TypeScript
    'react/react-in-jsx-scope': 'off',  // Not needed in React 17+
    'react-hooks/rules-of-hooks': 'error',
    'react-hooks/exhaustive-deps': 'warn',
    
    /* General rules */
    'no-console': ['warn', { allow: ['warn', 'error'] }],
    'prefer-const': 'error',
    'no-var': 'error',
    'eqeqeq': ['error', 'always'],
  },
};

üéØ Essential TypeScript-ESLint Rules

üö´ no-explicit-any

Prevents use of 'any' type

// ‚ùå Bad
function process(data: any) {}

// ‚úÖ Good
function process(data: unknown) {}
function process<T>(data: T) {}

‚è≥ no-floating-promises

Must handle promise results

// ‚ùå Bad
fetchData();

// ‚úÖ Good
await fetchData();
fetchData().catch(handleError);
void fetchData();

üîó prefer-optional-chain

Use optional chaining

// ‚ùå Bad
const name = user && user.profile 
  && user.profile.name;

// ‚úÖ Good
const name = user?.profile?.name;

üí´ consistent-type-imports

Separate type imports

// ‚ùå Bad
import { User } from './types';

// ‚úÖ Good
import type { User } from './types';

üìù Package.json Scripts

{
  "scripts": {
    "lint": "eslint . --ext .ts,.tsx",
    "lint:fix": "eslint . --ext .ts,.tsx --fix",
    "type-check": "tsc --noEmit",
    "check": "npm run type-check && npm run lint"
  }
}
üí° VS Code Integration: Install the ESLint extension and add to settings.json: 
{
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true
  },
  "eslint.validate": [

  ]

18.3 Monorepo Setup with TypeScript

üì¶ What is a Monorepo?

A monorepo is a single repository containing multiple packages or applications. This is common for design systems, shared libraries, or microservices architectures.

üí° Benefits:
  • Shared code and types across packages
  • Atomic commits across multiple packages
  • Simplified dependency management
  • Better code reuse

üèóÔ∏è Monorepo Structure
