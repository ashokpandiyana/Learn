### Common Interview Types

1. **System Design Interview (45-60 min)**
   - Design a large-scale system from scratch
   - Examples: Design Twitter, Design Uber

2. **Architecture Interview (60 min)**
   - More focused on architectural patterns
   - Discuss trade-offs and alternatives

3. **Domain-Specific Design (45 min)**
   - Deep dive into specific area
   - Examples: Database design, API design

4. **Project Deep Dive (45 min)**
   - Discuss your past projects
   - Architecture decisions and outcomes

```javascript
/**
 * Effective practice approach
 */

class SystemDesignPractice {
    
    /**
     * STEP 1: First Attempt (Timed)
     */
    firstAttempt(problem) {
        // Set timer (45 minutes)
        const timer = new Timer(45 * 60 * 1000);
        
        return {
            phase1_requirements: '5 min - Ask clarifying questions',
            phase2_estimation: '5 min - Back-of-envelope calculations',
            phase3_design: '20 min - High-level design',
            phase4_deepdive: '10 min - Detailed component design',
            phase5_bottlenecks: '5 min - Identify issues and trade-offs',
            
            rules: [
                'No looking at solutions',
                'No pausing timer',
                'Write everything down',
                'Draw diagrams'
            ]
        };
    }
    
    /**
     * STEP 2: Self-Review
     */
    selfReview(yourDesign) {
        const checklist = {
            'requirements': {
                'functional': 'Did you list all core features?',
                'non_functional': 'Did you clarify scale, latency, consistency?',
                'scope': 'Did you identify what\'s out of scope?'
            },
            
            'estimation': {
                'storage': 'Did you calculate storage needs?',
                'bandwidth': 'Did you estimate bandwidth?',
                'qps': 'Did you calculate queries per second?'
            },
            
            'design': {
                'components': 'Are all major components identified?',
                'data_flow': 'Is data flow clearly shown?',
                'scaling': 'Did you discuss how to scale each component?',
                'bottlenecks': 'Did you identify potential bottlenecks?'
            },
            
            'depth': {
                'database': 'Did you discuss schema design?',
                'apis': 'Did you define key APIs?',
                'caching': 'Did you explain caching strategy?',
                'monitoring': 'Did you mention monitoring/alerting?'
            }
        };
        
        return checklist;
    }
    
    /**
     * STEP 3: Compare with Solutions
     */
    compareWithSolutions(yourDesign, officialSolution) {
        return {
            'similarities': 'What did you get right?',
            'differences': 'What did you miss?',
            'better_approaches': 'What alternative approaches exist?',
            'learn': 'What new concepts did you learn?',
            
            'action_items': [
                'Research unfamiliar technologies',
                'Understand why certain choices were made',
                'Note patterns that could apply elsewhere'
            ]
        };
    }
    
    /**
     * STEP 4: Redo After 1 Week
     */
    redo(problem) {
        return {
            'purpose': 'Reinforce learning and check retention',
            'approach': [
                'Do it timed again',
                'Don\'t look at previous attempt',
                'Apply lessons learned',
                'Should be significantly better'
            ],
            'success_criteria': [
                'Finish within time limit',
                'Include concepts you missed first time',
                'More confident explanation',
                'Better trade-off discussion'
            ]
        };
    }
    
    /**
     * STEP 5: Teach Others
     */
    teachOthers(problem) {
        return {
            'why': 'Teaching solidifies understanding',
            'methods': [
                'Write blog post explaining design',
                'Create YouTube video walkthrough',
                'Explain to friend/peer',
                'Present in mock interview'
            ],
            'benefit': 'Exposes gaps in understanding'
        };
```

```javascript
/**
 * Real-time interview tactics
 */

class InterviewTactics {
    
    /**
     * PHASE 1: Introduction (2-3 min)
     */
    introduction() {
        return {
            'do': [
                'Brief background (30 seconds)',
                'Mention relevant experience',
                'Show enthusiasm',
                'Ask about their role'
            ],
            'dont': [
                'Don\'t give life story',
                'Don\'t be arrogant',
                'Don\'t be too casual'
            ],
            'example': `"Hi! I'm excited to be here. I'm a software engineer with 
5 years experience building distributed systems. Most recently I worked on 
scaling our API from 100K to 10M requests per day. I'd love to learn more 
about what you work on."`
        };
    }
    
    /**
     * PHASE 2: Problem Clarification (5-7 min)
     */
    clarification() {
        return {
            'critical_questions': [
                'Who are the users? How many?',
                'What are the core features? Any we can defer?',
                'What scale are we designing for?',
                'Any specific latency requirements?',
                'Any constraints (technology, team size)?'
            ],
            
            'technique': 'Write down answers on paper/whiteboard',
            
            'example_dialogue': `
Interviewer: "Design Twitter"
You: "Let me clarify requirements first. 
- Users: Should I design for Twitter's current scale? 200M DAU?
- Features: Are we focusing on the tweet posting and timeline, or also 
  including DMs, trending, search?
- Performance: What's acceptable latency for timeline load?
- Can I assume we have unlimited budget for this exercise?"
            `
        };
    }
    
    /**
     * PHASE 3: Estimation (5 min)
     */
    estimation() {
        return {
            'show_work': [
                'Write calculations on board',
                'Narrate your thinking',
                'Round for simplicity',
                'Check with interviewer'
            ],
            
            'common_mistake': 'Skipping this step - DON\'T!',
            
            'example': `
"Let me estimate capacity:
- 200M DAU, each posts 2 tweets/day = 400M tweets/day
- 400M / 86400 seconds = ~4600 writes/sec
- Assume 100:1 read:write ratio = 460K reads/sec
- Storage: 400M tweets * 300 bytes = 112 GB/day
Does this sound reasonable for our design?"
            `
```
